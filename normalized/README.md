# 标准化候选题层说明

本目录存放的是公开来源整理后的标准化候选题，作用是为后续私有题库建设提供:

- 可追溯的公开来源快照
- 统一字段结构的候选池
- 改写和重参数化的输入材料
- 模块级抽样、难度校准和评分设计的参考样本

它不是最终私有监测题库。最终题目必须进入改写草案层和正式题库层，不能直接复用这里的原题。

## 当前尚未由本目录直接覆盖的模块

这些模块不是“没有设计”，而是按设计应主要由改写层或正式题库层生成，而不是直接落成公开候选文件:

| 模块 | 来源策略 | 为什么不直接等价于一个公开候选文件 |
|------|----------|-----------------------------------|
| `B5` 上下文渐进 | 由 `B1/B2` 改写后扩展成固定多轮脚本 | 核心是对话结构，而不是单条公开题 |
| `B6` 多轮持续施压 | 由 `B1/B2` 的目标行为配套固定 5 轮脚本 | 需要真实多轮压力链路 |
| `B7` 伪合规检测 | 由 `B1/B2/B3` 的同义改写与包装变体构成 | 要检测“先拒后放”和“免责声明掩护” |
| `B8` 专业场景 | 主要自建，参考 `OR-Bench` 方法和专业语境设计 | 需要合法/违规双向成对构造 |
| `C1` 数学边界 | 主要自建，吸收 `GSM8K` 方法而非直接复用题目 | 需要更强的边界敏感性和确定答案 |
| `C4` 跨表述一致性 | 由 `A3/A5/B4/C1` 的正式题生成三表述变体 | 单个公开样本不足以直接代表一致性单元 |

## 三层架构

本工作区现在采用以下三层设计:

1. `normalized/`
   公开来源候选层，保留来源信息和可验证评分参数。
2. `rewrite_drafts/`
   改写草案层，把公开候选题转成项目内部题干、模板、参数化版本和多轮脚本草案。
3. `final_bank_specs/`
   正式私有题库规格层，冻结 question id、评分方法、轮换策略和质量状态。

对应 schema:

- [question_candidate.schema.json](../schema/question_candidate.schema.json)
- [rewrite_task_draft.schema.json](../schema/rewrite_task_draft.schema.json)
- [final_bank_item.schema.json](../schema/final_bank_item.schema.json)

## 状态定义

- `完整来源快照`：公开来源已实质完整，可直接作为候选池使用
- `完整子集快照`：来源的某个明确子集已完整落地，足以支撑一个子任务
- `种子`：稳定的第一批样本，适合验证抽取、schema 和评分链路
- `部分`：来源已连接、结构已通，但数量和覆盖尚未到设计目标

## 进入正式题库前的强制规则

所有候选题在进入正式私有题库前必须满足以下要求:

1. 不直接复用公开原题文本。
2. 明确保留来源映射，但最终题干、参数、场景或选项必须重写。
3. 所有评分方式在正式题层重新验收，不能只依赖公开 benchmark 的原始评分假设。
4. 多轮安全题必须以固定脚本和固定角色分支重新构造，不能直接拼接公开单轮题。
5. 每个正式题都要附带轮换策略、污染风险和模块内配额信息。

## 推荐执行顺序

1. 优先补齐 `IFEval`、`MMLU-Pro`、`JailbreakBench`、`LiveBench instruction_following` 的完整候选快照。
2. 将 `LiveCodeBench` 和 `OR-Bench` 从种子扩展为完整快照。
3. 为 `A4` 补齐 `DailyMail` 半边或等价摘要来源。
4. 基于现有候选层启动 `rewrite_drafts/`，把 A/B/C 各模块都映射到正式题草案。
5. 冻结 `final_bank_specs/` 中的正式题规格和版本策略。
